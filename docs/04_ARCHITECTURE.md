# noet Architecture: Техническая архитектура

> Этот документ описывает полную техническую архитектуру noet по слоям. Каждый слой включает: ответственность, требования, варианты реализации, рекомендуемый выбор и обоснование.

---

## Архитектурные принципы

Прежде чем описывать слои, фиксируем принципы, которые определяют все решения:

1. **Протокол над платформой** — каждый слой — это протокол, не продукт
2. **Слои независимы** — замена одного слоя не требует переписывания других
3. **Полнота через избыточность** — система работает, пока работает часть узлов
4. **Самоудостоверяемость** — данные проверяются без центрального сервера
5. **Постепенная сложность** — простые случаи просты, сложные возможны
6. **P2P-фIRST** — децентрализованный путь предпочтительнее централизованного
7. **Отказоустойчивость** — каждый компонент предполагает отказ других

## 1. Identity Layer

### Ответственность
- Создание и управление криптографическими идентичностями
- Подпись и верификация данных
- Криптографические доказательства принадлежности
- Восстановление идентичности

### Требования
- Не требует сервера для создания идентичности
- Работает в offline
- Поддерживает мультиподпись и социальное восстановление
- Совместим с существующими системами (Nostr, DID)
- Квантово-устойчивые подписи (перспектива)

### Варианты реализации

| Технология | Децентрализация | Зрелость | Проблемы |
|---|---|---|---|
| **Nostr-ключи (secp256k1 Schnorr)** | Полная | Высокая | Нет структуры, просто ключи |
| **W3C DID** | Зависит от метода | Стандартизирована | Сложность, разные методы |
| **ATProto DID (did:plc, did:web)** | Частичная | Средняя | did:plc централизован |
| **Ethereum-адреса** | Зависит от сети | Высокая | Зависимость от блокчейна |
| **Ed25519 ключи (SSB)** | Полная | Высокая | Меньше экосистема |

### Рекомендуемый выбор: Nostr-ключи + W3C DID (гибрид)

**Обоснование:**

Nostr демонстрирует, что простейшая возможная система идентичности (пара ключей secp256k1) работает масштабно. Но noet требует большего: восстановление, делегирование, ротацию. Поэтому:

- **Базовый уровень** — Nostr-ключи (secp256k1 Schnorr): просто, работает, огромная экосистема
- **Расширенный уровень** — W3C DID для структурированной идентичности с документами
- **Восстановление** — социальное восстановление через NIP-46 (Nostr Connect) + пороговая схема
- **Делегирование** — NIP-26 (delegated event signing) для делегирования прав без передачи ключей

**Почему не ATProto:** did:plc контролируется Bluesky — единая точка отказа. Это противоречит фундаментальным принципам noet.

**Почему не чистый DID:** W3C DID — хороший стандарт, но конкретные методы (did:ethr, did:web) привносят зависимости. Nostr-ключи как базовый уровень дают простоту и независимость.

## 2. Discovery Layer

### Ответственность
- Поиск узлов по идентификаторам
- Обнаружение контента
- Обнаружение сообществ
- Peer discovery в P2P-сети

### Требования
- Работает без центрального сервера
- Устойчив к цензуре (не один узел отвечает за поиск)
- Масштабируется до миллионов узлов
- Быстрый lookup (секунды, не минуты)

### Варианты реализации

| Технология | Тип | Зрелость | Масштаб |
|---|---|---|---|
| **Kademlia DHT (libp2p)** | Полностью децентрализованный | Очень высокая | Миллионы узлов (BitTorrent) |
| **HyperDHT (Holepunch)** | Полностью децентрализованный | Средняя | Тысячи узлов |
| **Nostr-релеи** | Федеративный | Высокая | Зависит от релея |
| **ATProto Relays/BGS** | Централизованный | Средняя | Ограничен Bluesky |
| **DNS + DNSSEC** | Централизованный | Высокая | Глобальный, но цензурируем |

### Рекомендуемый выбор: libp2p DHT (Kademlia)

**Обоснование:**

libp2p Kademlia DHT — battle-tested в BitTorrent (миллионы узлов), IPFS, Ethereum. Обеспечивает:

- **O(log n)** lookup в сети из n узлов
- **Цензуроустойчивость** — нет центрального реестра
- **Редупликацию** — данные хранятся на k ближайших узлах
- **Интеграцию** — libp2p предоставляет единый интерфейс для DHT, pub/sub, транспорта

**Для noet:**
- Узлы анонсируют себя в DHT по публичным ключам
- Сообщества анонсируются по своим идентификаторам
- Контент адресуется по CID (content identifier)
- Peer discovery работает через DHT + mDNS (в локальных сетях) + bootstrap-списки

**Риски:**
- Sybil-атаки на DHT — митигация через proof-of-work или stake для node ID
- Eclipse-атаки — митигация через защиту routing table, multi-path lookups
- Churn — узлы часто отключаются, митигация через агрессивное переподключение

## 3. Networking Layer

### Ответственность
- Установление соединений между узлами
- NAT traversal
- Шифрование транспорта
- Мультиплексирование протоколов
- Устойчивость к разрывам

### Требования
- Работает за NAT
- Работает в браузере
- Шифрование по умолчанию
- Низкая задержка
- Поддержка разных транспортов (TCP, UDP, QUIC, WebRTC)

### Варианты реализации

| Технология | P2P | Браузер | NAT | Шифрование |
|---|---|---|---|---|
| **libp2p** | Да | Да (js-libp2p) | Да (circuit relay, AutoNAT) | Noise + TLS |
| **Holepunch** | Да | Нет (нужен Pear Runtime) | Да (UDP hole punching) | Noise |
| **WebRTC** | Да | Нативно | Да (ICE, STUN, TURN) | DTLS + SRTP |
| **yggdrasil** | Да | Нет | Да | Curve25519 |
| **I2P** | Да | Нет | Да | Garlic routing |

### Рекомендуемый выбор: libp2p

**Обоснование:**

libp2p — золотой стандарт P2P-нетворкинга. Обоснование:

- **Множественные реализации** — Go, Rust, JavaScript, Python. Не привязка к одному языку.
- **Производственное использование** — Ethereum, Filecoin, IPFS, Polkadot — миллиарды в обеспечении
- **Модульность** — можно менять транспорты, шифрование, discovery независимо
- **QUIC** — современный транспорт с встроенным шифрованием, мультиплексированием, быстрым handshake
- **WebRTC** — поддержка браузерных узлов
- **Circuit Relay** — обход NAT через промежуточные узлы
- **AutoNAT** — автоматическое определение собственной доступности
- **GossipSub** — эффективный pub/sub для messaging layer

**Конкретный стек:**
- Транспорт: QUIC (основной), WebRTC (браузеры), TCP (fallback)
- Шифрование: Noise Protocol Framework
- NAT: AutoNAT + Circuit Relay v2
- Мультиплексирование: Yamux + QUIC streams
- Peer Routing: Kademlia DHT
- Content Routing: Kademlia DHT + IPFS delegated routing

## 4. Messaging Layer

### Ответственность
- Передача сообщений между участниками
- Pub/sub для тем/каналов
- Гарантия доставки (best-effort)
- Упорядочивание сообщений в контексте

### Требования
- Децентрализованная маршрутизация
- Масштабируется до тысяч участников в теме
- Низкая задержка (для online-участников)
- Offline-доставка (через хранение на узлах)

### Варианты реализации

| Технология | Тип | Масштаб | Зрелость |
|---|---|---|---|
| **libp2p GossipSub** | Полностью децентрализованный | 100K+ топиков | Очень высокая (Ethereum) |
| **Nostr-релеи** | Федеративный | Зависит от релея | Высокая |
| **Matrix-серверы** | Федеративный | Зависит от сервера | Высокая |
| **SSB (gossip)** | Полностью децентрализованный | Сотни (ограничение) | Средняя |
| **ATProto (relay)** | Централизованный | Очень высокий | Средняя |

### Рекомендуемый выбор: libp2p GossipSub

**Обоснование:**

GossipSub — production-grade в Ethereum (десятки тысяч узлов). Обеспечивает:

- **Mesh propagation** — сообщения распространяются через mesh-граф подписчиков
- **Fan-out** — для публикаторов, не подписанных на топик
- **Prune/backoff** — адаптация топологии под нагрузку
- **Сообщения подписываются** — верификация источника
- **Score-based peer management** — защита от вредоносных узлов

**Для noet:**
- Каждое сообщество — отдельный GossipSub-топик (или иерархия топиков)
- noet распространяются через GossipSub
- Подписчики получают сообщения в реальном времени
- Узлы хранят сообщения для offline-участников

**Почему не Nostr-релеи:** Релеи — федеративные точки отказа. Владелец релея контролирует, какие события принимать. noet требует полной децентрализации на уровне протокола.

## 5. Storage Layer

### Ответственность
- Хранение noet (контентных единиц)
- Хранение медиафайлов (изображения, видео, документы)
- Адресация по содержимому (CID)
- Репликация и доступность
- Долгосрочное архивирование

### Требования
- Адресация по содержимому (не по местоположению)
- Самоудостоверяемость (данные проверяются по CID)
- Репликация на несколько узлов
- Работа в offline (локальная копия)
- Эффективность для мелких объектов (noet) и крупных (медиа)

### Варианты реализации

| Технология | CID | Репликация | Мелкие объекты | Крупные файлы |
|---|---|---|---|---|
| **IPFS + Helia** | Да | DHT-based | Хорошо | Хорошо |
| **Kubo (go-ipfs)** | Да | DHT-based | Хорошо | Хорошо |
| **Hypercore** | Да (по ключу) | P2P sync | Отлично (append-only) | Через Hyperblobs |
| **Hyperdrive** | Да | P2P sync | Хорошо | Отлично (файловая система) |
| **Arweave** | Да | Permanent | Дорого | Дорого |
| **BitTorrent** | Да (infohash) | Через swarm | Нет | Отлично |
| **Syncthing** | Нет (путь) | P2P sync | Хорошо | Хорошо |

### Рекомендуемый выбор: IPFS + Hypercore (гибрид)

**Обоснование:**

Ни одна технология не покрывает все кейсы. Гибридный подход:

**Для noet (мелкие подписанные объекты): Hypercore**

- Append-only log идеален для ленты сообщений
- Каждое сообщество = один Hypercore (или дерево Hypercores)
- Sparse replication — узел скачивает только нужные диапазоны
- Криптографическая верификация каждого блока
- Быстрая синхронция

**Для медиафайлов: IPFS**

- CID-адресация работает на глобальном уровне
- DHT-based discovery — файлы находятся без центрального индекса
- Широкая экосистема (шлюзы, pinning-сервисы)
- BitSwap для эффективного обмена

**Для архивирования: опционально Arweave**

- Для критически важных данных, которые должны существовать десятилетия
- One-time payment для permanent storage
- Не обязательно для базового функционирования

**Интеграция:**

```
noet (Hypercore append-only log)
├── content: текст, метаданные, ссылки
└── media_refs: [CID1, CID2, ...] → IPFS

Медиа (IPFS)
├── CID = хэш содержимого
├── Хранится на узлах сообщества
└── Доступен через IPFS DHT

Архив (опционально Arweave)
├── Транзакция = хэш + метаданные
├── Permanent, но медленный
└── Для конституций, важных документов
```

**Почему не чистый IPFS:** Для append-only логов сообщений IPFS неоптимален. Каждое обновление требует нового CID. Hypercore разработан именно для этого кейса.

**Почему не чистый Hypercore:** Hypercore — лог, не файловая система. Для медиафайлов IPFS предоставляет лучшую экосистему.

## 6. Replication Layer

### Ответственность
- Синхронизация состояния между узлами
- Конфликт-фри редактирование (CRDT)
- Оффлайн-работа с последующей синхронизацией
- Консистентность в распределённой среде

### Требования
- Работа offline с последующим merge
- Автоматическое разрешение конфликтов
- Низкая задержка синхронизации
- Масштабируемость

### Варианты реализации

| Технология | Тип | Зрелость | Производительность |
|---|---|---|---|
| **Yjs** | CRDT | Очень высокая | 26K-156K ops/sec |
| **Automerge** | CRDT | Высокая (2.0) | 600ms для 260K ops |
| **OrbitDB** | CRDT + IPFS | Средняя | Зависит от IPFS |
| **Fireproof** | CRDT | Средняя | Хорошая |
| **Diamond Types** | CRDT | Средняя | Очень высокая |
| **Operational Transform** | Централизованный | Высокая | Требует центральный сервер |

### Рекомендуемый выбор: Yjs

**Обоснование:**

Yjs — зрелейшая CRDT-библиотека, используется в production (Figma использует CRDT-подход, Yjs — стандарт для web):

- **Экстремальная производительность** — 26K-156K операций в секунду
- **Богатая структура данных** — Text, Array, Map, XML
- **Провайдеры** — WebRTC, WebSocket, IndexedDB, все необходимые транспорты
- **Awareness protocol** — курсоры, присутствие, undo/redo
- **Маленький размер** — ~20KB gzip

**Для noet:**
- Совместное редактирование wiki-страниц, документов
- Синхронизация состояния чата
- Real-time коллаборация
- Offline-first: работа с локальной копией, sync при подключении

**Automerge 2.0** — отличная альтернатива для специфических кейсов, особенно где важна портируемость (Rust-core работает везде).

## 7. Realtime Layer (Voice/Video)

### Ответственность
- Голосовые звонки
- Видеозвонки
- Групповые конференции
- Экранный обмен
- Стриминг

### Требования
- Минимальная задержка
- Работа без центрального сервера (где возможно)
- Масштабируемость до сотен участников
- Шифрование

### Анализ технологий

#### 7.1. Huddle01

**Что это:** DePIN (Decentralized Physical Infrastructure Network) для RTC, построенный на Arbitrum.

**Архитектура:**
- Медиа-трафик маршрутизируется через Media Nodes (майнеры с токеном HUDL)
- Сигнализация через блокчейн-координацию
- Цель: заменить централизованные SFU распределённой сетью

**Где децентрализован:**
- Узлы медиа-роутинга распределены (множество Media Nodes)
- Нет единого владельца инфраструктуры
- Токен-стимулы для запуска узлов

**Где централизован:**
- Сигнализационный слой зависит от блокчейна Arbitrum
- Координация узлов требует смарт-контрактов
- Токеномика создаёт экономическую централизацию (крупные стейкеры)
- Кодовая база контролируется компанией Huddle01
- Привязка к конкретному L2 (Arbitrum) — точка отказа

**Вывод:** Huddle01 — интересный эксперимент, но не подходит для noet из-за:
- Зависимости от блокчейна
- Токеномических рисков
- Недостаточной проверенности (новый проект)
- Экономической модели, несовместимой с долговечностью

#### 7.2. Waterbus

**Что это:** Open-source WebRTC SFU, написанный на Rust.

**Архитектура:**
- Distributed SFU с Redis-backed signaling
- Поддержка SFU и P2P гибридной топологии
- LL-HLS streaming, simulcast, virtual background

**Где децентрализован:**
- Код open source — кто угодно может запустить свой SFU
- Distributed architecture с несколькими нодами

**Где централизован:**
- По сути — это просто open-source Zoom. Каждый инстанс централизован
- Нет P2P-маршрутизации между разными инстансами
- Redis — единая точка отказа для конкретного инстанса

**Вывод:** Waterbus — хороший SFU для self-hosted видеоконференций, но не является децентрализованным решением. Может использоваться как компонент, но не как realtime-слой noet.

#### 7.3. Mumble

**Что это:** Open-source voice chat, созданный для геймеров. Существует с 2005 года.

**Архитектура:**
- Клиент-сервер (Mumble client + Murmur server)
- Низкая задержка (10-30ms round-trip)
- Opus кодек
- Positional audio
- ACL-based permissions

**Где децентрализован:**
- Любой может запустить Murmur сервер
- Код open source, протокол документирован
- Нет зависимости от какой-либо компании

**Где централизован:**
- Каждый Murmur сервер — центральная точка для своих пользователей
- Нет federation между серверами
- Нет P2P (всё идёт через сервер)

**Вывод:** Mumble — надёжный, проверенный временем инструмент для голосового чата. Для noet может служить fallback или опцией для сообществ, предпочитающих серверную модель. Но это не децентрализованное решение.

#### 7.4. WebRTC P2P + SFU Fallback

**Архитектура для noet:**

```
1:1 звонок (два участника):
├── Direct WebRTC P2P (без сервера)
├── Сигнализация через libp2p (Signal protocol)
└── Полностью децентрализован

Малый групповой чат (3-8 участников):
├── Full mesh WebRTC (каждый с каждым)
├── Или выбор "супер-пира" (участник с лучшим соединением)
└── Почти децентрализован

Крупная конференция (9+ участников):
├── Выборный SFU (один из участников или выделенный узел)
├── Или federated SFU (несколько узлов координируются)
└── Гибрид: P2P + опциональный SFU
```

**Рекомендуемый выбор: WebRTC P2P + опциональный self-hosted SFU**

**Обоснование:**

Истинно децентрализованное видео для групп >8 человек — нерешённая проблема во всей индустрии. Физические ограничения (bandwidth каждого пира) делают полный mesh невозможным для больших групп.

noet решает это прагматично:

1. **1:1 и малые группы** — полный P2P через WebRTC, сигнализация через libp2p
2. **Крупные группы** — участники могут выбрать SFU-узел (как релей). Это может быть:
   - Один из участников с хорошим bandwidth
   - Выделенный узел сообщества (как Mumble-сервер, но контролируется сообществом)
   - Внешний сервис (Waterbus, LiveKit, Jitsi) — если сообщество предпочитает
3. **Федерация SFU** — несколько узлов ретранслируют между собой для географического масштабирования

**Критически важно:** Выбор SFU принадлежит сообществу. Это не центральная инфраструктура noet. Каждое сообщество решает само, как обеспечивать видео для крупных конференций.

## 8. Media Layer (Streaming)

### Ответственность
- Доставка видео/аудио контента множеству зрителей
- Адаптивный битрейт
- Запись и архивирование
- Live streaming

### Варианты реализации

| Технология | Децентрализация | Масштаб | Зрелость |
|---|---|---|---|
| **WebTorrent** | P2P (BitTorrent в браузере) | До тысяч | Высокая |
| **HLS P2P** | Гибрид (CDN + P2P) | Очень высокий | Средняя |
| **IPFS Live** | P2P | Средний | Эксперимент |
| **LiveKit Egress** | Централизованный | Очень высокий | Высокая |

### Рекомендуемый выбор: WebTorrent

**Обоснование:**

WebTorrent — единственная проверенная P2P-технология для стриминга, работающая в браузере:

- **Streaming торренты** — видео начинает воспроизводиться до полной загрузки
- **WebRTC data channels** — работает без плагинов
- **BitTorrent DHT** — миллионы узлов, проверенная масштабируемость
- **Интеграция с <video>** — прямое воспроизведение в браузере
- **Seeding** — зрители становятся сидами, разгружая оригинальный источник

**Для noet:**
- Записанные конференции, стримы, подкасты — распространяются через WebTorrent
- magnet-ссылки или infohash публикуются как noet
- Зрители скачивают через WebTorrent (встроен в клиент noet)
- Для live-стримов — WebRTC-based решения (временная сегментация)

## 9. Governance Layer

### Ответственность
- Хранение конституций сообществ
- Ведение голосований
- Подсчёт результатов
- Исполнение решений
- Экономика сообществ (если есть)

### Требования
- Прозрачность (все голоса проверяемы)
- Неподкупность (нельзя подменить результат)
- Доступность (работает даже при спорах)
- Гибкость (разные сообщества — разные правила)

### Варианты реализации

| Технология | Тип | Зрелость | Применение |
|---|---|---|---|
| **Snapshot** | Федеративный | Высокая | Off-chain голосование, on-chain исполнение |
| **Gnosis Safe** | Мультиподпись | Высокая | Управление тresaury |
| **Moloch DAO** | On-chain | Высокая | Простое управление |
| **Aragon** | On-chain | Средняя | Сложные организации |
| **Custom (noet-native)** | Off-chain + опционально on-chain | Новая | Максимальная гибкость |

### Рекомендуемый выбор: noet-native + опциональная интеграция со Snapshot/Gnosis

**Обоснование:**

Гovernance noet не требует блокчейна для базового функционирования:

- **Голосование** — подписанные noet с типом "vote", подсчёт на узлах
- **Конституция** — хранится как Hypercore, версионируется
- **Репутационный вес** — вычисляется локально каждым узлом
- **Исполнение** — автоматизировано через консенсус узлов

**Для экономических функций** (treasury, staking, токены сообществ) — опциональная интеграция:
- **Snapshot** для голосований с привязкой к токенам
- **Gnosis Safe** для мультиподписных кошельков сообществ
- **Любой EVM/SVM** для smart contracts

**Ключевой принцип:** Governance ядра noet не зависит от блокчейна. Блокчейн — опциональный слой для тех сообществ, которым нужна on-chain экономика.

## 10. Runtime Layer

### Ответственность
- Среда выполнения клиентского приложения
- Доступ к файловой системе, сети, криптографии
- Кросс-платформенность (Windows, macOS, Linux, Mobile, Web)
- Безопасность (sandbox, permissions)

### Варианты реализации

| Технология | Платформы | Размер | Безопасность | Зрелость |
|---|---|---|---|---|
| **Tauri** | Win/Mac/Linux/iOS/Android | 3-10 MB | Высокая (Rust, capabilities) | Высокая (v2) |
| **Electron** | Win/Mac/Linux | 120-200 MB | Средняя | Очень высокая |
| **Browser Extension** | Chrome/Firefox/Safari | Малый | Ограничена средой | Высокая |
| **Browser (PWA)** | Все с браузером | Нет установки | Ограничена sandbox | Высокая |
| **Native Desktop** | Платформа-специфичный | Минимальный | Высокая | Зависит |
| **Native Mobile** | iOS/Android | Минимальный | Высокая | Зависит |
| **Flutter** | Все | 10-20 MB | Хорошая | Высокая |

### Рекомендуемый выбор: Tauri (основной) + Browser/PWA (альтернатива)

**Обоснование:**

**Tauri v2** — оптимальный баланс для noet:

- **3-10 MB** против 120-200 MB у Electron (30-60x меньше)
- **Rust backend** — память-безопасность, производительность, нативный доступ к системе
- **Capability-based security** — приложение имеет доступ только к разрешённым ресурсам
- **Кросс-платформа** — Windows, macOS, Linux, iOS, Android из одной кодовой базы
- **Native WebView** — использует системный браузерный движок
- **WASM support** — криптографические операции можно выполнять на Rust/WASM

**Почему не Electron:**
- Слишком тяжёлый (150+ MB)
- Высокое потребление RAM
- Каждое приложение тащит свой Chromium — избыточность
- Большая поверхность атаки

**Почему не Browser Extension:**
- Зависимость от магазинов (Chrome Web Store может удалить)
- Ограниченный доступ к системе
- Не работает на мобильных устройствах

**Почему Browser/PWA — важная альтернатива:**
- Мгновенный доступ без установки
- Независимость от магазинов приложений
- Достаточно для чтения, лёгкого общения
- Для полнофункционального использования — Tauri-клиент

**Стратегия:**
- **Основной клиент** — Tauri (максимальная функциональность, лучшая производительность)
- **Лёгкий клиент** — Browser/PWA (доступность, быстрый старт)
- **Мобильный** — Tauri v2 (iOS/Android)

## 11. Applications Layer

### Ответственность
- Пользовательские интерфейсы
- Специфическая функциональность (форумы, чаты, wiki)
- Интеграция со слоями ниже
- Плагин-архитектура

### Архитектура

noet не предписывает конкретные приложения. Вместо этого предоставляет:

**Core SDK** — библиотека для взаимодействия со всеми слоями:
```
noet-sdk (TypeScript/Rust)
├── identity/        — управление ключами, подпись
├── discovery/       — поиск узлов, контента, сообществ
├── storage/         — чтение/запись noet, медиа
├── replication/     — CRDT синхронизация
├── messaging/       — pub/sub, подписки
├── governance/      — голосование, конституции
└── realtime/        — WebRTC, голос, видео
```

**Plugin Architecture:**
```
Каждое приложение = набор плагинов к базовому клиенту
├── Форум: thread-viewer, reply-composer, moderation-tools
├── Чат: room-list, message-view, typing-indicator
├── Wiki: editor (Yjs-based), version-history, diff-viewer
├── Стрим: WebTorrent-player, chat-overlay
└── DAO: proposal-creator, voting-interface, treasury-view
```

**Application Registry:**
- Приложения распространяются через децентрализованный registry (IPFS-based)
- Каждое приложение подписано разработчиком
- Пользователи выбирают, какие приложения установить
- Нет единого магазина приложений

## Сравнение с текущим стеком noet

| Компонент | Текущий стек | Рекомендуемый стек | Изменение |
|---|---|---|---|
| Identity | Nostr | Nostr + W3C DID | Расширение |
| Storage | IPFS + Kubo | IPFS + Hypercore | Гибридизация |
| Naming | Registry → CID | DHT (libp2p) + опциональные DNS | Замена |
| Discovery | Resolver | libp2p DHT | Замена |
| Client | Browser Extension | Tauri + Browser | Замена |
| Networking | Не указан | libp2p | Добавление |
| Messaging | Не указан | GossipSub | Добавление |
| Realtime | Не указан | WebRTC P2P + опц. SFU | Добавление |
| Replication | Не указан | Yjs CRDT | Добавление |
| Governance | Gnosis Safe + Snapshot | noet-native + опционально | Упрощение |

**Ключевые изменения:**
1. **libp2p как фундамент** — единый P2P-стек вместо набора отдельных решений
2. **Hypercore для сообщений** — append-only logs лучше подходят для лент, чем raw IPFS
3. **Tauri вместо Browser Extension** — независимость от браузерных магазинов, лучшая производительность
4. **CRDT для синхронизации** — offline-first редактирование
5. **Governance без обязательного блокчейна** — проще, дешевле, быстрее
