# SCZ — аудит проекта: расхождения и разделение узел/зона

Ресёрч текущего состояния против docs (§03 System Model, §04 Architecture, §02 Philosophy)
и наших планов. Главный принцип: **узел = протокол, зона = приложения ПОВЕРХ протокола.**
Приложения (лента, мессенджер, сообщества, вики, звонок) это не части узла, а веб-надстройка.

## Что должно жить ПОЛНОСТЬЮ в приложении (узел = бэкбон)

Узел это сетевой участник и хранитель, а не «ещё одна вкладка». Полностью в узле:

1. **Личность** — ключи, мульти-аккаунт, подпись BIP340, шифрование (NIP-04, цель NIP-44),
   восстановление (NIP-46/threshold), делегирование (NIP-26). Приватный ключ не покидает узел.
2. **Связь (СЕТЬ)** — узел держит соединения с сетью (сейчас Nostr-реле, цель libp2p
   GossipSub/DHT). Узел публикует, подписывается, кэширует, синхронит в фоне. Зона спрашивает
   данные У УЗЛА, а не лезет в реле сама.
3. **Хранение** — узел = IPFS-нода (iroh/Kubo): пиннинг и раздача медиа в P2P. Append-only
   логи (Hypercore, цель). Локальный кэш/индекс (SQLite).
4. **Резолв имён и индекс** — узел строит и держит индекс имён/профилей/страниц, отдаёт зоне.
5. **Самообновление** — tauri-updater.
6. **Репутация** — вычисление по проверяемым событиям на стороне узла (а не наспех в браузере).

## Что живёт в зоне (приложения поверх)

Через `window.nostr`/`window.noet` и API узла: лента, мессенджер (каналы, ЛС), сообщества,
вики, Люди, домены/сайты, поиск, управление, UI звонка. Это UX, не протокол. Любой может
сделать свой клиент-зону поверх того же узла.

## Расхождения с доками/планами/философией (от важного к мелкому)

| # | Расхождение | Где правильно | Серьёзность |
|---|---|---|---|
| 1 | **Связь живёт в браузере, не в узле.** Зона сама открывает WebSocket к реле. Узел не является сетевым участником, он только подписывает и отдаёт статику. | §04: узел = бэкбон/связь. Перенести соединения в узел, зона ходит через `/api/query|publish|subscribe`. Тогда фон, персистентность, и миграция на libp2p без переписывания приложений. | КРИТ |
| 2 | **Нет P2P.** Сеть = публичные реле (костыль §11). libp2p/DHT/GossipSub не реализованы. | §04.2-4 целевое. Роадмап. | Высокая |
| 3 | **Хранилище только на чтение.** IPFS-шлюзы для чтения + прокси; записи/пиннинга нет, медиа только инлайн. | §04.5: узел = IPFS-нода (запись/пин). | Высокая |
| 4 | **Репутация примитивна.** Одно число, глобально, по числу своих событий, считается в браузере. | §03.1.3: многомерная (вклад/конструктивность/экспертиза/управление), per-community, сигналы (лайки/флаги/награды), сибил-защита. | Высокая |
| 5 | **Сообщества не сущности.** Нет ролей, конституции, состояния членства (CRDT), форка. Только темы. | §03.1.2: сообщество с конституцией, ролями, форком. | Высокая |
| 6 | **Управление базовое.** Предложения+голоса есть, но без привязки к конституции, исполнения решений, разделения властей, арбитража. | §03.1.7, §06. | Средняя |
| 7 | **Сибил-защита слабая.** Порог репутации = счётчик постов; бот накрутит. | §03.3: PoW/stake для node ID, сибил-устойчивый DHT, охлаждение. | Средняя |
| 8 | **DM на NIP-04** (устаревший). | Цель NIP-44 (ChaCha20+HKDF). | Средняя |
| 9 | **Нет восстановления/делегирования** (NIP-46/26). | §04.1. | Средняя |
| 10 | **Нет Hypercore/Yjs** (append-логи, совместное редактирование). | §04.5-6. Роадмап. | Средняя |
| 11 | **Индекс поиска строится в браузере** каждую сессию. | Индекс на стороне узла. | Низкая |

## Очередь выравнивания (по ценности)

1. **Связь в узел** (#1): узел держит реле-соединения, зона ходит через API узла. Это
   разблокирует фон, персистентность и переход на libp2p без переписывания приложений.
2. **Узел = IPFS-нода** (#3): запись/пиннинг медиа P2P (iroh/Kubo), без аккаунтов.
3. **Репутация по модели §03.1.3** (#4) + сибил-защита (#7).
4. **Сообщества как сущности** (#5): роли, конституция, членство, форк (Yjs состояние).
5. **Управление** (#6) привязать к конституции и исполнению.
6. **NIP-44 для DM** (#8), восстановление/делегирование (#9).
7. **libp2p/Hypercore** (#2, #10): целевой сетевой/хранилищный слой §04. Многомесячно.
