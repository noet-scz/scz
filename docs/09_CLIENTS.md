# SCZ Clients: Анализ клиентских платформ

> SCZ — это протокол, но люди взаимодействуют с ним через клиенты. Этот документ анализирует варианты клиентских платформ и выбирает главный клиент.

---

## Требования к клиенту SCZ

### Функциональные
- [ ] Криптография: генерация ключей, подпись, верификация
- [ ] Сеть: P2P-соединения через libp2p
- [ ] Хранение: локальная база данных (noet, медиа)
- [ ] UI: отображение контента, создание noet
- [ ] Governance: голосование, создание предложений
- [ ] Realtime: WebRTC для голоса/видео
- [ ] Media: проигрывание видео/аудио

### Нефункциональные
- [ ] Малый размер (желательно < 20 MB)
- [ ] Быстрый запуск (< 1 секунды)
- [ ] Низкое потребление ресурсов
- [ ] Безопасность: sandbox, ограниченные права
- [ ] Кросс-платформа: Windows, macOS, Linux, iOS, Android
- [ ] Независимость от магазинов приложений
- [ ] Работа offline

---

## Анализ вариантов

### 1. Browser Extension (Chrome/Firefox/Safari)

**Как работает:**
- Расширение к браузеру, внедряющее функциональность SCZ
- Доступ к страницам, API браузера
- Background script для P2P-соединений

**Плюсы:**
- Быстрый доступ (иконка в браузере)
- Может интегрироваться с веб-сайтами
- Лёгкое обновление через магазин расширений
- Малый размер

**Минусы:**
- **Зависимость от магазинов** — Chrome Web Store может удалить расширение
- **Ограниченный API** — Service Workers не могут держать постоянные P2P-соединения
- **Нет UDP** — WebRTC работает, но DHT с ограничениями
- **Нет background-работы** — браузер останавливает Service Worker
- **Только desktop** — мобильные браузеры не поддерживают расширения (кроме Firefox Android)
- **Производительность** — JavaScript в ограниченной среде

**Оценка:**

| Критерий | Оценка |
|---|---|
| Децентрализация | Низкая (зависимость от Google/Mozilla) |
| P2P-соединения | Ограниченные |
| UX | Средняя |
| Ресурсы | Низкие |
| Кросс-платформа | Нет (только desktop) |
| Безопасность | Средняя |
| Оффлайн | Ограниченный |
| **Итого** | **Не рекомендуется как основной клиент** |

---

### 2. Browser Runtime (PWA/Web App)

**Как работает:**
- Веб-приложение, открываемое в браузере
- Service Workers для кэширования и offline-работы
- WebRTC для P2P
- IndexedDB для локального хранения

**Плюсы:**
- **Независимость от магазинов** — доступно по URL
- **Мгновенный доступ** — открыл ссылку, начал пользоваться
- **Кросс-платформа** — любое устройство с браузером
- **Обновления мгновенные** — нет процесса одобрения
- **WebRTC нативно** — поддерживается в браузере
- **Работает offline** (с Service Worker)

**Минусы:**
- **P2P ограничения** — WebRTC требует signaling, нет raw UDP
- **Нет TCP/QUIC напрямую** — только через WebSocket или WebRTC
- **Background sync ограничен** — браузер может остановить Service Worker
- **Хранение ограничено** — IndexedDB лимиты (зависят от браузера)
- **DHT ограничен** — js-libp2p в браузере работает через WebSocket/WebRTC, не UDP
- **Push-уведомления** — требуют центрального push-сервиса

**Оценка:**

| Критерий | Оценка |
|---|---|
| Децентрализация | Средняя (зависимость от браузера, но не от магазина) |
| P2P-соединения | Частичные (через WebRTC) |
| UX | Высокая (привычный веб-интерфейс) |
| Ресурсы | Низкие |
| Кросс-платформа | Высокая |
| Безопасность | Средняя (sandbox браузера) |
| Оффлайн | Частичный |
| **Итого** | **Рекомендуется как лёгкий клиент** |

---

### 3. Tauri

**Как работает:**
- Desktop/mobile приложение с Rust backend
- Frontend: HTML/CSS/JS (WebView)
- Backend: Rust (системный доступ)
- Компилируется в нативный бинарник

**Плюсы:**
- **Малый размер** — 3-10 MB (против 120-200 MB у Electron)
- **Быстрый запуск** — < 200 ms
- **Rust backend** — безопасность памяти, высокая производительность
- **Capability-based security** — приложение запрашивает только нужные права
- **Полный системный доступ** — UDP, TCP, файловая система (через Rust)
- **libp2p нативно** — rust-libp2p или go-libp2p (через FFI)
- **Кросс-платформа** — Windows, macOS, Linux, iOS, Android (v2)
- **Независимость** — нет магазина, распространение через сайт/торренты
- **Оффлайн** — полнофункциональный
- **Background** — работает как полноценное приложение

**Минусы:**
- **Меньше экосистемы** чем Electron (но растёт быстро)
- **Требуется Rust** для backend-разработки (но JS API покрывает 80%)
- **WebView inconsistencies** — разные движки на разных ОС (решается тестированием)
- **Mobile менее зрелый** — v2 вышел в 2024, но активно развивается

**Оценка:**

| Критерий | Оценка |
|---|---|
| Децентрализация | Высокая (нет зависимости от магазинов/платформ) |
| P2P-соединения | Полные (нативный libp2p) |
| UX | Высокая (web-технологии для UI) |
| Ресурсы | Очень низкие |
| Кросс-платформа | Высокая (desktop + mobile в v2) |
| Безопасность | Высокая (Rust + capabilities) |
| Оффлайн | Полный |
| **Итого** | **Рекомендуется как основной клиент** |

### 4. Electron

**Как работает:**
- Desktop приложение = Chromium + Node.js + ваш код
- Frontend и backend — JavaScript
- Полный доступ к системе через Node.js API

**Плюсы:**
- **Огромная экосистема** — тысячи приложений (VS Code, Discord, Slack)
- **Зрелость** — существует с 2013 года
- **JavaScript everywhere** — один язык для всего
- **Отладка** — Chrome DevTools
- **Богатые API** — доступ к системе через Node.js

**Минусы:**
- **Огромный размер** — 120-200 MB
- **Высокое потребление RAM** — 150-400 MB в idle
- **Медленный запуск** — 2-5 секунд
- **Каждое приложение = Chromium** — избыточность
- **Большая поверхность атаки** — полный Node.js + Chromium
- **Только desktop** — нет mobile

**Оценка:**

| Критерий | Оценка |
|---|---|
| Децентрализация | Средняя (независимость от магазина, но от Google Chromium) |
| P2P-соединения | Полные (нативный Node.js) |
| UX | Высокая |
| Ресурсы | Очень высокие |
| Кросс-платформа | Только desktop |
| Безопасность | Средняя (большая поверхность) |
| Оффлайн | Полный |
| **Итого** | **Fallback, но не основной выбор** |

### 5. Native Desktop (Rust/Qt/Swift)

**Как работает:**
- Полностью нативное приложение для каждой платформы
- Нет WebView, нет JavaScript

**Плюсы:**
- **Максимальная производительность**
- **Минимальный размер**
- **Нативный UX** — выглядит как приложение ОС
- **Полный системный доступ**

**Минусы:**
- **Три кодовые базы** — Windows, macOS, Linux отдельно
- **Сложнее найти разработчиков** (особенно Rust/Qt)
- **Медленнее разработка**
- **Нет web-экосистемы** — компоненты с нуля

**Оценка:**

| Критерий | Оценка |
|---|---|
| Децентрализация | Высокая |
| P2P-соединения | Полные |
| UX | Высокая (нативная) |
| Ресурсы | Очень низкие |
| Кросс-платформа | Низкая (3 кодовые базы) |
| Безопасность | Высокая |
| Оффлайн | Полный |
| **Итого** | **Для premium-клиентов, не основной** |

### 6. Mobile Native (React Native / Flutter)

**Как работает:**
- Кросс-платформенная разработка для iOS/Android
- React Native: JavaScript + нативные компоненты
- Flutter: Dart + собственный рендеринг

**Плюсы:**
- **Кросс-платформа** — одна кодовая база для iOS и Android
- **Нативный UX** — выглядит как нативное приложение
- **Хорошая экосистема**
- **Публикация в App Store / Google Play**

**Минусы:**
- **Зависимость от Apple/Google** — могут удалить приложение
- **P2P ограничен** — iOS/Android ограничивают background networking
- **Push-уведомления** — требуют Firebase/APNs
- **Хранение** — ограничения sandbox

**Оценка:**

| Критерий | Оценка |
|---|---|
| Децентрализация | Низкая (зависимость от Apple/Google) |
| P2P-соединения | Ограниченные (background) |
| UX | Высокая (нативная) |
| Ресурсы | Средние |
| Кросс-платформа | Высокая (одна кодовая база) |
| Безопасность | Средняя |
| Оффлайн | Частичный |
| **Итого** | **Дополнительный, не основной** |

## Рекомендация: Главный клиент — Tauri

### Почему Tauri:

1. **Размер 3-10 MB** — пользователи не боятся скачивать
2. **Запуск < 200ms** — приложение открывается мгновенно
3. **Rust backend** — безопасность, производительность, доступ к системе
4. **libp2p через Rust** — полноценный P2P-узел
5. **Кросс-платформа** — desktop сейчас, mobile в v2
6. **Независимость** — распространение через сайт, GitHub, торренты
7. **Capability-based security** — приложение запрашивает только нужные права
8. **Web-технологии для UI** — React/Vue/Svelte, огромная экосистема
9. **Оффлайн-first** — работает без интернета, синхронизируется при подключении

### Архитектура Tauri-клиента SCZ:

```
Tauri App (3-10 MB)
├── Frontend (React/Vue + TypeScript)
│   ├── UI: чаты, каналы, профили
│   ├── State: Zustand/Redux
│   └── Components: design system
├── Backend (Rust)
│   ├── libp2p: P2P, DHT, GossipSub
│   ├── Hypercore: sync logs
│   ├── IPFS: media storage/retrieval
│   ├── Yjs: CRDT sync
│   ├── Crypto: key management, signing
│   └── Storage: SQLite for local cache
└── Bridge (Tauri Commands + Events)
    ├── Async commands: Rust → JS
    └── Events: JS → Rust
```

### Альтернативные клиенты (параллельно):

| Клиент | Назначение | Приоритет |
|---|---|---|
| **PWA** | Мгновенный доступ без установки | Высокий (параллельно) |
| **Tauri Desktop** | Основной, полнофункциональный | Высокий (основной) |
| **Tauri Mobile** | iOS/Android (v2) | Средний |
| **Terminal/CLI** | Для разработчиков, скриптов | Низкий |

### Распространение (без магазинов):

- **Основной:** Скачивание с официального сайта
- **Зеркала:** GitHub Releases, IPFS, торренты
- **Checksums:** Криптографические хэши для проверки
- **Auto-update:** Через Tauri updater (с проверкой подписей)
- **Форки:** Любой может форкнуть клиент и распространять

## Заключение

Главный клиент SCZ — **Tauri**. Он даёт оптимальный баланс:

- Малый размер (3-10 MB) — не отпугивает пользователей
- Полная функциональность — нативный P2P, offline, всё
- Независимость — не требует магазинов
- Кросс-платформа — desktop сейчас, mobile в ближайшем будущем
- Безопасность — Rust + capability model

PWA — важная альтернатива для мгновенного доступа, но с P2P-ограничениями.

Electron — fallback для сценариев, где нужна максимальная совместимость web-экосистемы.
