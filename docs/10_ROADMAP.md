# noet Roadmap: Путь от 0 до 100,000+

> Этот документ описывает поэтапный путь развития noet. Каждый этап включает цели, метрики, технические задачи и риски.

---

## Принципы roadmap

1. **Работает → Красиво** — сначала функциональность, потом полировка
2. **Один шаг за раз** — не пытаться построить всё сразу
3. **Фидбек от пользователей** — каждая фаза даёт информацию для следующей
4. **Долговечность > скорость** — архитектурные решения не торопятся
5. **Сообщество = продукт** — рост сообщества важнее роста кода

---

## Фаза 0: Протокол (0 пользователей)

**Длительность:** 6-12 месяцев

**Цель:** Работающие спецификации и реализация базовых протоколов.

### Задачи

- [ ] **Identity protocol v1**
  - Генерация secp256k1 ключей
  - Schnorr подпись/верификация
  - NIP-46 (Nostr Connect) для делегирования
  - Социальное восстановление (пороговая схема)

- [ ] **Networking protocol v1**
  - libp2p интеграция (rust-libp2p)
  - QUIC транспорт
  - Kademlia DHT
  - GossipSub pub/sub
  - NAT traversal (AutoNAT + Circuit Relay)

- [ ] **Storage protocol v1**
  - Hypercore интеграция
  - Append-only logs для сообщений
  - Sparse replication
  - IPFS для медиа

- [ ] **noet data model v1**
  - Спецификация типов noet
  - Связи между noet
  - CRDT для редактируемых типов (Yjs)

- [ ] **Governance protocol v1**
  - Конституционный формат
  - Proposal workflow
  - Голосование (подписанные noet)
  - Подсчёт результатов

- [ ] **Tauri client v0.1**
  - Базовый UI
  - Генерация ключей
  - Подключение к сети
  - Публикация/чтение noet

### Метрики завершения
- [ ] 5+ разработчиков понимают протоколы
- [ ] 3+ независимые реализации клиентов (хотя бы минимальные)
- [ ] Спецификации опубликованы и рецензированы

### Риски
| Риск | Митигация |
|---|---|
| Сложность libp2p | Начать с js-libp2p, мигрировать на rust |
| Гиперкор не стабилен | Fallback на IPFS MFS |
| Недостаток разработчиков | Баунти, хакатоны, Nostr-экосистема |

## Фаза 1: Пионеры (0-100 пользователей)

**Длительность:** 3-6 месяцев

**Цель:** Первые реальные сообщества, отладка протоколов, фидбек.

### Задачи

- [ ] **Tauri client v0.5**
  - Стабильный P2P (соединения, синхронизация)
  - Создание/присоединение к сообществам
  - Лента сообщений (Hypercore-based)
  - Профили, ключи, backup

- [ ] **Первые сообщества**
  - 3-5 пионерских сообществ (разработчики, энтузиасты)
  - Тестирование governance workflow
  - Отладка конституций

- [ ] **Модуль realtime (базовый)**
  - WebRTC 1:1 звонки
  - Сигнализация через libp2p

- [ ] **Документация**
  - Протоколы задокументированы
  - SDK для разработчиков
  - Примеры приложений

- [ ] **Стабилизация**
  - Багфиксы из реального использования
  - Оптимизация производительности
  - Улучшение UX

### Метрики
- [ ] 100+ зарегистрированных идентичностей
- [ ] 5+ активных сообществ
- [ ] 50%+ retention (пользователи возвращаются)
- [ ] 0 критических багов в протоколах

### Что измеряем
- Удержание пользователей (retention)
- Количество сообщений в день
- Время синхронизации
- Количество багов

## Фаза 2: Ранние последователи (100-1000)

**Длительность:** 6-9 месяцев

**Цель:** Рост через сарафанное радио, первые "убийственные фичи", стабильность.

### Задачи

- [ ] **Tauri client v1.0**
  - Полнофункциональный Discord/Reddit-like опыт
  - Голосовые каналы (WebRTC mesh)
  - File sharing (IPFS)
  - Поиск по сообщениям
  - Нотификации

- [ ] **PWA клиент (лёгкий)**
  - Базовый функционал без установки
  - Чтение, написание, голосование

- [ ] **Governance v2**
  - Шаблоны конституций
  - UI для создания proposals
  - Голосование с репутацией
  - Модераторские инструменты

- [ ] **Realtime v2**
  - Групповые звонки до 8 человек (mesh)
  - Запись звонков (IPFS)
  - Screen sharing

- [ ] **Onboarding**
  - Простое создание идентичности
  - Поиск сообществ
  - Туториал

- [ ] **Перформанс**
  - Синхронизация < 1 секунды для новых сообщений
  - Запуск клиента < 500ms
  - Потребление RAM < 100MB

### Метрики
- [ ] 1000+ идентичностей
- [ ] 50+ сообществ
- [ ] 1000+ сообщений в день
- [ ] DAU/MAU > 30%

### Ключевые решения
- Как работает governance на практике?
- Как пользователи находят сообщества?
- Что мотивирует запускать узлы?

## Фаза 3: Рост (1000-10000)

**Длительность:** 9-12 месяцев

**Цель:** Масштабирование, разнообразие приложений, экономика.

### Задачи

- [ ] **SDK для разработчиков**
  - noet-sdk (TypeScript/Rust)
  - Документация
  - Примеры: бот, интеграция, кастомный клиент

- [ ] **Плагин-архитектура**
  - Расширение клиента плагинами
  - Plugin marketplace (децентрализованный)
  - Темы, боты, интеграции

- [ ] **Специализированные клиенты**
  - Forum-клиент (Reddit-like)
  - Chat-клиент (Telegram-like)
  - Stream-клиент (Twitch-like)

- [ ] **Realtime v3**
  - Групповые звонки до 50 (выборный SFU)
  - Streaming (WebTorrent-based)
  - Записи, клипы

- [ ] **Экономика (опциональная)**
  - Интеграция со Snapshot для token-based governance
  - Gnosis Safe для treasury
  - Lightning Network для донатов

- [ ] **Reputation system v1**
  - Многомерная репутация
  - Визуализация
  - История изменений

- [ ] **Mobile (Tauri v2)**
  - iOS клиент
  - Android клиент

### Метрики
- [ ] 10,000+ идентичностей
- [ ] 500+ сообществ
- [ ] 10,000+ сообщений в день
- [ ] 10+ сторонних клиентов/плагинов
- [ ] 100+ публичных узлов

### Риски
| Риск | Митигация |
|---|---|
| Спам при росте | Reputation gating, PoW |
| Недостаток узлов | Инструменты для easy node setup |
| DDoS | Resource manager, community-run relays |
| Governance атаки | Репутационные пороги, таймеры |

## Фаза 4: Масштаб (10000-100000)

**Длительность:** 12-18 месяцев

**Цель:** Масштабирование до 100K пользователей, зрелая экосистема.

### Задачи

- [ ] **Масштабирование Hypercore**
  - Шардирование (одно сообщество = несколько Hypercores)
  - Эффективная синхронция (только diff)
  - Архивные узлы для старых данных

- [ ] **Масштабирование DHT**
  - Sybil-resistant DHT (PoW для node IDs)
  - Оптимизация routing table
  - Bootstrap diversification

- [ ] **Search & Discovery**
  - Децентрализованный поиск сообществ
  - Content discovery (не алгоритмический)
  - Теги, категории

- [ ] **Inter-community features**
  - Кросс-постинг
  - Federated identity (репутация переносится)
  - Shared channels

- [ ] **Enterprise features**
  - Self-hosted узлы для компаний
  - Интеграция с LDAP/Active Directory
  - Audit logs

- [ ] **Security hardening**
  - Formal verification критических компонентов
  - Bug bounty программа
  - Security audit

- [ ] **Governance v3**
  - Жидкая демократия
  - Межсообщественные суды
  - Автоматизированное исполнение

### Метрики
- [ ] 100,000+ идентичностей
- [ ] 5000+ сообществ
- [ ] 100,000+ сообщений в день
- [ ] 1000+ публичных узлов
- [ ] 50+ сторонних клиентов

---

## Фаза 5: Экосистема (100000+)

**Длительность:** 18+ месяцев

**Цель:** noet как инфраструктурный слой, тысячи сообществ, экономическая устойчивость.

### Задачи

- [ ] **Protocol standardization**
  - IETF-style RFC для ключевых протоколов
  - Множественные независимые реализации
  - Interoperability testing

- [ ] **Economic sustainability**
  - Сообщества финансируют разработку
  - Grants, donations, optional tokens
  - Независимость от венчурного капитала

- [ ] **Post-quantum cryptography**
  - Исследование PQ-подписей
  - Гибридные схемы
  - Миграционный план

- [ ] **Advanced applications**
  - Децентрализованный маркетплейс приложений
  - AI-ассистенты (локальные, не облачные)
  - VR/AR интеграция

- [ ] **Governance autonomy**
  - Основная команда не нужна
  - Полностью сообщество-управляемая разработка
  - BDFL → выборный technical council

### Метрики
- [ ] 1,000,000+ идентичностей
- [ ] 50,000+ сообществ
- [ ] 1,000,000+ сообщений в день
- [ ] 10,000+ узлов
- [ ] noet незаметна как протокол (как DNS или TCP)

## Заключение

Roadmap — не догма. Каждая фаза даёт фидбек, который может изменить приоритеты. Ключевые принципы:

1. **Протоколы первичны** — без стабильных протоколов нет долговечности
2. **Сообщество важнее кода** — люди приходят за людьми, не за технологиями
3. **Форк — не провал** — раскол сообщества лучше, чем диктатура
4. **noet переживёт нас** — архитектура важнее скорости

Путь от 0 до 100,000 — 3-4 года. Путь от 100,000 к инфраструктурному слою — ещё 2-3 года. Это марафон, не спринт.
