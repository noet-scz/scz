# SCZ Governance: Модель власти

> SCZ — это не только протокол, но и политическая система. Этот документ описывает, как сообщества управляют собой, как принимаются решения, как предотвращается захват власти, и как система переживает своих основателей.

---

## 1. Философия управления

### 1.1. Почему управление важно

Без управления цифровое сообщество либо погружается в анархию (захват троллями), либо превращается в диктатуру основателя. SCZ ставит целью:

- Дать сообществам инструменты для самоуправления
- Предотвратить захват власти любой группой
- Обеспечить право меньшинства на форк без потери данных
- Создать механизмы, работающие без основателей

### 1.2. Принципы управления SCZ

1. **Конституционность** — правила записаны и автоматизированы
2. **Разделение властей** — законодательная, исполнительная, судебная
3. **Федерализм** — решения на самом низком уровне
4. **Право на выход** — форк всегда возможен
5. **Постепенная децентрализация** — от диктатуры основателя к демократии
6. **Суверенитет** — каждое сообщество определяет свои правила

## 2. Конституция сообщества

### 2.1. Что такое конституция

Конституция — это машиночитаемый документ, определяющий:

- Цели и ценности сообщества
- Права участников
- Правила поведения
- Структуру управления
- Процедуры принятия решений
- Механизмы изменения конституции
- Процедуры форка

### 2.2. Структура конституции

```yaml
constitution:
  meta:
    version: 1.0.0
    created: <timestamp>
    amended: <timestamp>
    
  principles:
    - <фундаментальные ценности>
    
  rights:
    participant:
      - right_to_speech
      - right_to_exit
      - right_to_fork
      - right_to_data
    minority:
      - protection_from_majority
      - right_to_secede
      
  rules:
    - <правила поведения>
    
  governance:
    legislature:
      - <как принимаются правила>
    executive:
      - <как применяются правила>
    judicial:
      - <как разрешаются споры>
      
  amendment:
    procedure: <процедура изменения>
    quorum: <минимальное участие>
    threshold: <требуемое большинство>
    
  fork:
    procedure: <права и процедура форка>
```

### 2.3. Каталог конституционных шаблонов

SCZ предоставляет набор проверенных шаблонов:

| Шаблон | Описание | Подходит для |
|---|---|---|
| **Диктатура основателя** | Полная власть создателя | Ранняя стадия, 1-10 человек |
| **Технократия** | Власть у экспертов (по репутации) | Технические сообщества |
| **Прямая демократия** | Каждый голосует по всем вопросам | Малые сообщества (<50) |
| **Представительная** | Выборные делегаты | Средние сообщества (50-500) |
| **Жидкая демократия** | Делегирование голосов | Большие сообщества (>500) |
| **Консенсус** | Решения только единогласно | Малые, когезивные группы |
| **Совет старейшин** | Власть у наиболее репутабельных | Зрелые сообщества |
| **Корпоративная** | Акционеры (токены) голосуют | Экономически ориентированные |

## 3. Механизмы принятия решений

### 3.1. Жизненный цикл предложения (proposal)

```
[Создание] → [Обсуждение] → [Уточнение] → [Голосование] → [Подсчёт] → [Исполнение]
```

**Создание:** Любой участник с минимальным порогом репутации создаёт proposal-noet

**Обсуждение (Discussion Phase):**
- Период 3-7 дней
- Участники обсуждают в комментариях
- Автор может вносить правки (новые версии proposal)
- Модераторы помечают дубли, оффтопик

**Уточнение (Refinement Phase):**
- 1-3 дня
- Финальная версия proposal фиксируется
- Больше изменений нет

**Голосование (Voting Phase):**
- 3-7 дней
- Участники голосуют: yes / no / abstain
- Вес голоса = функция от репутации (не 1 человек = 1 голос)

**Подсчёт:**
- Автоматический, проверяемый любым узлом
- Результат фиксируется в noet

**Исполнение:**
- Если принято — изменения применяются автоматически (через код) или вручную делегатами

### 3.2. Вес голоса

В SCZ вес голоса не равен 1 человек = 1 голос. Варианты:

| Модель | Формула | Плюсы | Минусы |
|---|---|---|---|
| **Репутационная** | вес = log(репутация) | Учитывает вклад | Барьер для новичков |
| **Квадратичная** | вес = √репутация | Баланс | Сложность |
| **Жидкая** | вес = репутация, но делегируется | Гибкость | Сложность, коалиции |
| **Временная** | вес = репутация × время_в_сообществе | Учитывает вовлечённость | Барьер |
| **Многомерная** | разные веса по разным темам | Экспертиза | Сложность |

**Рекомендация:** Репутационная с квадратичным корнем для малых сообществ, жидкая демократия для больших.

### 3.3. Кворум и пороги

| Тип решения | Кворум | Порог | Период |
|---|---|---|---|
| **Обычное правило** | 20% участников | 50%+1 | 3 дня |
| **Важное правило** | 40% участников | 60%+1 | 5 дней |
| **Изменение конституции** | 60% участников | 75%+1 | 7 дней |
| **Форк** | Не требуется | Инициатива меньшинства | — |
| **Назначение модератора** | 30% участников | 50%+1 | 3 дня |
| **Исключение участника** | 40% участников | 66%+1 | 5 дней |

## 4. Разделение властей

### 4.1. Законодательная власть

**Кто:** Все участники (через голосование)
**Что:** Принятие и изменение правил, конституции
**Как:** Proposal + voting

### 4.2. Исполнительная власть

**Кто:** Модераторы (выборные или назначенные)
**Что:** Применение правил к конкретным случаям
- Скрытие/показ контента
- Удаление спама
- Бан участников (временный или постоянный)
- Управление каналами

**Ограничения:**
- Модераторы действуют в рамках конституции
- Любое действие можно оспорить
- Модераторы подотчётны и могут быть сняты

### 4.3. Судебная власть

**Кто:** Арбитражные советы (выборные, ротация)
**Что:** Разрешение споров
- Апелляции на решения модераторов
- Конфликты между участниками
- Толкование конституции

**Процедура:**
1. Участник подаёт апелляцию (appeal-noet)
2. Случайный выбор N арбитров из пула (с учётом конфликта интересов)
3. Арбитры рассматривают, голосуют
4. Решение окончательно (или подлежит апелляции в крайних случаях)

### 4.4. Аудиторская власть

**Кто:** Любой участник (прозрачность)
**Что:** Проверка корректности работы системы
- Верификация подсчёта голосов
- Проверка, что модераторы действуют по правилам
- Аудит финансов (если есть treasury)

## 5. Предотвращение захвата власти

### 5.1. Модели захвата и защиты

| Атака | Механизм | Защита SCZ |
|---|---|---|
| **Захват большинством** | Накрутка аккаунтов, Sybil-атака | Репутационный порог, квадратичное голосование |
| **Захват богатыми** | Покупка токенов/репутации | Непередаваемость репутации, ceiling на вес голоса |
| **Захват модераторами** | Сговор модераторов | Ротация, случайный выбор, право апелляции |
| **Захват основателем** | Основатель не отпускает власть | Конституционные таймеры, обязательная децентрализация |
| **Захват разработчиками** | Команда контролирует код | Open source, форк всегда возможен |
| **Захват через токены** | Whales контролируют голосование | Ceiling, многомерность, не только токены |

### 5.2. Механизмы защиты

**Конституционные таймеры:**
- Новое сообщество может начинаться с диктатуры основателя
- Но конституция должна содержать таймер: через N дней/участников управление переходит к сообществу
- Это обязательное требование для шаблона "Диктатура основателя"

**Ротация модераторов:**
- Модераторы избираются на срок (например, 6 месяцев)
- Переизбрание не более N раз подряд
- Случайный выбор из кандидатов с достаточной репутацией

**Потолок власти (Power Ceiling):**
- Максимальный вес одного голоса (например, 5% от общего)
- Максимальное количество модераторских действий в день
- Максимальная длительность полномочий

**Право на форк:**
- Любое меньшинство может форкнуть
- Форк не требует разрешения
- Форк сохраняет все данные
- Это "ядерная опция", которая предотвращает любой захват

## 6. Право на форк

### 6.1. Что такое форк в SCZ

Форк — создание нового сообщества на базе существующего с копией всех данных. Оба сообщества существуют независимо.

### 6.2. Процедура форка

```
1. Инициация
   └── Группа участников создаёт fork-proposal-noet
   └── Указывают причину, изменения в конституции
   
2. Объявление
   └── Уведомление всего сообщества
   └── Период размышления (например, 7 дней)
   
3. Выбор
   └── Каждый участник решает: остаться, перейти, или быть в обоих
   └── Выбор фиксируется noet
   
4. Создание
   └── Новое сообщество создаётся с копией данных
   └── Новая конституция применяется
   
5. Разделение
   └── Оба сообщества живут независимо
   └── Связь может поддерживаться или нет
```

### 6.3. Права при форке

- Любое меньшинство (даже 1 человек) имеет право форкнуть
- Не требуется согласие оригинального сообщества
- Данные копируются полностью (контент, репутация на момент форка)
- Форкующие не могут говорить от имени оригинала
- Оригинал не может препятствовать форку

### 6.4. Форк как эволюция

В SCZ форк — не катастрофа, а естественный процесс:

- Конкуренция форков ведёт к лучшим решениям
- "Голосование ногами" — участники выбирают лучшее сообщество
- Успешные форки доказывают свою состоятельность
- Неуспешные форки отмирают (данные сохраняются, но активность нулевая)

## 7. Жизнь после основателей

### 7.1. Проблема Буссардиона

В римской республике законы Сервия Туллия считались неприкосновенными, хотя никто не помнил их автора. SCZ ставит целью то же: сообщество переживает основателя.

### 7.2. Механизмы переживания

**Документирование:**
- Вся история, все решения, все дебаты — в noet
- Новые участники могут изучить историю
- Нет зависимости от памяти живых людей

**Автоматизация:**
- Возможно, управление автоматизируется
- Рутинные решения — через proposal workflow
- Сложные — через выборных представителей

**Ротация:**
- Ключевые роли не привязаны к личностям
- Система позволяет передавать полномочия
- Нет единой точки отказа

**Форк как страховка:**
- Если сообщество застопорилось после ухода основателя — форк
- Форк запускает новую жизнь с теми же данными
- Естественный отбор форков → лучшие практики

### 7.3. Стадии жизни сообщества

| Стадия | Характер | Управление | Основной риск |
|---|---|---|---|
| **Зарождение** | Основатель + несколько человек | Диктатура основателя | Основатель пропадает |
| **Рост** | 10-100 участников | Основатель + первые модераторы | Конфликт видений |
| **Зрелость** | 100-10000 | Выборное управление | Бюрократия |
| **Автономия** | 10000+ | Полная децентрализация | Застой |
| **Эволюция** | Форки, слияния | Конкуренция моделей | Распад |

## 8. Межсообщественные отношения

### 8.1. Федерация

Сообщества могут устанавливать федеративные связи:

- **Взаимное признание репутации** — участники с репутацией в A имеют стартовую репутацию в B
- **Обмен контентом** — кросс-постинг, репосты между сообществами
- **Совместное управление** — совместные комитеты для общих вопросов
- **Общая инфраструктура** — совместные узлы, shared SFU для видео

### 8.2. Конфликты между сообществами

SCZ не решает конфликты между сообществами. Нет единого арбитра. Варианты:

- **Независимость** — каждое сообщество решает само
- **Федерация** — договорённости через представителей
- **Форк** — если конфликт неразрешим внутри, часть уходит
- **"Холодная война"** — оба сообщества существуют, не общаясь

## 9. Инструменты управления

### 9.1. On-chain vs Off-chain

SCZ governance преимущественно **off-chain**:

**Почему off-chain:**
- Не требует газа
- Быстрее (нет ожидания блока)
- Не привязано к конкретному блокчейну
- Масштабируется лучше

**Когда on-chain:**
- Управление treasury (токены, криптовалюта)
- Механизмы staking/slashing
- Доказательство существования (timestamping)
- Межсообщественные контракты

### 9.2. Инструментарий

| Инструмент | Применение | Тип |
|---|---|---|
| **Proposal-noet** | Создание предложений | SCZ-native |
| **Vote-noet** | Голосование | SCZ-native |
| **Reputation engine** | Подсчёт репутации | SCZ-native |
| **Snapshot** | Token-based голосование | On-chain (opt) |
| **Gnosis Safe** | Мультиподпись treasury | On-chain (opt) |
| **Aragon** | Сложные организации | On-chain (opt) |
| **Kleros** | Децентрализованный суд | On-chain (opt) |

## 11. Заключение

Управление в SCZ — это не бюрократия. Это **инфраструктура свободы**. Хорошие механизмы управления позволяют сообществам:

- Существовать десятилетия
- Переживать основателей
- Адаптироваться к изменениям
- Защищаться от захвата
- Эволюционировать через форки

Плохое управление убивает сообщества. SCZ создаёт условия, при которых хорошее управление не зависит от харизматичного лидера. Оно встроено в протокол.
