# SCZ Stack: Рекомендуемый технологический стек

> Этот документ фиксирует финальный рекомендуемый стек для SCZ. Каждая технология оценена по критериям: зрелость, риски, альтернативы.

|---|---|---|---|---|

## Детальное обоснование каждого выбора

### Identity: Nostr keys + W3C DID

**Почему не чистый W3C DID:**
- DID — фреймворк, не конкретная реализация
- Методы did:web, did:plc требуют центральных компонентов
- Сложность спецификации препятствует долговечности

**Почему не ATProto DID:**
- did:plc контролируется одной организацией (Bluesky)
- Это точка отказа, которую SCZ не может принять
- Зависимость от доменов для handle

**Почему Nostr keys:**
- Простейшая возможная система: одна ключевая пара = одна личность
- Миллионы существующих идентичностей
- Экосистема клиентов, релеев, инструментов
- Простота = долговечность

**Расширение до W3C DID:**
- Для сообществ, которым нужна структурированная идентичность
- Документ DID содержит публичные ключи, сервисы, методы аутентификации
- Резолвинг через DHT или well-known endpoints

### Networking: libp2p

**Почему не Holepunch:**
- Holepunch (Pear Runtime) — JavaScript-only
- Меньше экосистема, меньше реализаций
- Зависимость от конкретной компании

**Почему не yggdrasil / I2P:**
- Они — сетевой уровень (как VPN), не application networking
- Добавляют слой абстракции, увеличивают задержку
- Не оптимизированы для pub/sub, DHT, content routing

**Почему libp2p:**
- 10+ реализаций на разных языках
- Проверен в production с миллиардами долларов (Ethereum, Filecoin)
- Модульная архитектура: можно заменить компоненты
- Активная разработка (300+ разработчиков/месяц)
- GossipSub, DHT, QUIC, WebRTC — всё в одном фреймворке

### Storage: Hypercore + IPFS

**Почему гибрид:**
- Hypercore оптимален для append-only логов (лента сообщений)
- IPFS оптимален для медиафайлов (CID-адресация, DHT discovery)
- У каждого — своя сильная зона

**Почему не чистый IPFS:**
- IPFS DAG — не append-only log
- Каждое обновление требует нового CID
- Нет встроенной криптографической верификации истории
- Hypercore разработан именно для этого use case

**Почему не Arweave для основного хранения:**
- Слишком дорого для часто меняющихся данных
- Оптимален для архивов, не для операционных данных
- Зависимость от токена и экономической модели

### Replication: Yjs

**Почему не Automerge:**
- Automerge 2.0 значительно улучшился (Rust core)
- Но Yjs всё ещё производительнее для типичных кейсов
- Yjs имеет больше провайдеров и интеграций

**Почему не Operational Transform:**
- OT требует центрального сервера
- Не работает в offline-first сценариях
- Сложнее в реализации

### Client: Tauri

**Почему не Electron:**
- 120-200 MB vs 3-10 MB
- 150-400 MB RAM vs 40-80 MB
- Каждое приложение = отдельный Chromium
- Большая поверхность атаки

**Почему не Flutter:**
- Отличный для mobile, но для desktop — меньше экосистемы
- Не использует web-технологии (сложнее найти разработчиков)
- Нет нативного доступа к web-стеку

**Почему Tauri:**
- Использует системный WebView (не bundled Chromium)
- Rust backend — безопасность и производительность
- Capability-based security model
- Мобильная поддержка в v2
- Быстрорастущая экосистема

### Governance: SCZ-native

**Почему не on-chain:**
- Блокчейн добавляет задержку, стоимость, сложность
- Не нужен для базового голосования и подсчёта
- Off-chain подписанные noet достаточны для 95% кейсов

**Почему опциональная интеграция:**
- Для сообществ с treasury — нужен блокчейн
- Snapshot — зрелый инструмент для token-based voting
- Gnosis Safe — зрелый инструмент для мультиподписи
- Но это слой приложений, не ядра

## Риски и митигации

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| Уязвимость в libp2p | Низкая | Высокое | Мониторинг security advisories, multi-implementation |
| Квантовый компьютер ломает крипто | Средняя (10-20 лет) | Катастрофическое | Roadmap на пост-квантовые подписи (Lamport, SPHINCS+) |
| Tauri не набирает массу | Средкая | Среднее | Fallback на Electron, кодовая база портативна |
| Hypercore заброшен | Средняя | Среднее | IPFS как fallback, форк возможен (open source) |
| GossipSub не масштабируется | Низкая | Высокое | Эмпирически проверен в Ethereum, есть fallback стратегии |
| IPFS gateway цензура | Средняя | Среднее | DHT-based routing, не зависимость от gateway |
| WebRTC меняет спецификацию | Низкая | Низкое | Абстракция слоя, libp2p адаптируется |
| Отсутствие мобильных разработчиков | Средкая | Среднее | PWA для mobile, Tauri v2 для нативных |

## Стратегия технологической эволюции

SCZ не привязана к конкретным версиям технологий. Стратегия:

1. **Абстракция** — каждый слой имеет чёткий интерфейс
2. **Изоляция** — изменение одного слоя не ломает другие
3. **Мониторинг** — отслеживание зрелости и безопасности зависимостей
4. **Консерватизм** — не обновляться на bleeding edge
5. **Диверсификация** — где возможно, поддержка нескольких реализаций
6. **Форки** — open source зависимости могут быть форкнуты

**Горизонт планирования:**

- **1-2 года** — текущий стек, стабилизация
- **3-5 лет** — оценка пост-квантовых подписей, возможная миграция
- **5-10 лет** — полная замена любого компонента возможна без разрушения системы
- **10+ лет** — система существует как протокол, технологии подменяются
