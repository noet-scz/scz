# SCZ — план реализации возможностей

Делаем НЕ «приложение», а **возможности зоны** (протокол + инфраструктура). Узел даёт
личность, связь, самообновление; всё остальное это веб-возможности, отдаваемые узлом в
браузер. Каждый пункт это способность системы из docs (§03 System Model, §08 Applications),
а не экран. Фаза-0 прагматична (Nostr-реле + IPFS), цель по §04 (libp2p/Hypercore/CRDT).

Статус: ✅ есть · 🟡 частично · ⬜ нет.

## 1. Личность (Identity) — §03.1.5
- ✅ Ключ secp256k1, подпись BIP340 в узле, без раскрытия приватного ключа.
- ✅ Мульти-аккаунт: создать/импортировать/переключить, активный виден (в окне узла).
- ✅ Профиль (имя, аватар, о себе), аватар через прокси узла.
- ✅ Тег/хэндл (основное имя), отдельно от ника, редактируется.
- ⬜ Восстановление: социальное (NIP-46/threshold) и мультиподпись.
- ⬜ Делегирование прав (NIP-26): постинг от имени без выдачи ключа.
- ⬜ Уровни доверия (аноним → заявленный → аттестованный → верифицированный, NIP-05).

## 2. Имена и домены (Naming) — расширение/§08
- ✅ Занять имя (claim 31111), резолв serverless по ранней заявке.
- ✅ Опубликовать под именем сайт/страницу/игру (page 31002), открыть на весь экран.
- ✅ Мост window.nostr/window.noet в опубликованный контент (игры/приложения работают).
- 🟡 Поиск по именам/профилям/страницам (есть, без удалений/рейтинга).
- ⬜ Поддомены (владелец base владеет *.base), передача/освобождение имени.
- ⬜ Медиа сайтов в IPFS (не только инлайн-HTML): загрузка файлов, картинок.

## 3. Сообщения (Messaging) — §08.1-2
- 🟡 Общая лента + каналы (kind 1 + тег темы). Базово.
- ⬜ Личные сообщения (DM) с E2EE (NIP-44), отдельный экран, список диалогов.
- ⬜ Каналы 1-to-many (только автор пишет) и группы (много авторов) как сущности.
- ⬜ Вложения (файлы/картинки через IPFS), голосовые сообщения (ogg/opus в IPFS).
- ⬜ Прочитано/непрочитано, уведомления (тумблер уже есть, доставки нет).

## 4. Сообщества (Communities) — §03.1.2
- 🟡 Пространство как тема (раньше Space 31000). Сейчас только каналы.
- ⬜ Сообщество как сущность: id, вступление (открытое/заявка/инвайт/голосование).
- ⬜ Роли (админ/модератор/участник) в состоянии сообщества (CRDT/Yjs цель).
- ⬜ Конституция (правила) как версионируемый документ.
- ⬜ Форк сообщества (копия данных, своя конституция) — фундамент §03.

## 5. Контент (noet) — §03.1.4
- 🟡 note/reply/article/wiki как события; вики была в расширении (порт нужен в зону).
- ⬜ Полные типы: thread, proposal, vote, media, reaction, badge.
- ⬜ Связи графа: reply/quote/fork/version/support/refute.
- ⬜ Вики в зоне (страницы по ключу, [[ссылки]], удаление) — порт из расширения.

## 6. Хранение (Storage) — §04.5
- 🟡 Мелкое = подписанные события на реле; медиа = IPFS-шлюзы (чтение), прокси узла.
- ⬜ Загрузка в IPFS из узла (свой Kubo/Helia или pinning), а не только чтение.
- ⬜ Целевое: Hypercore append-only логи лент, sparse-репликация (§04.5, роадмап).

## 7. Репутация (Reputation) — §03.1.3
- 🟡 Локальный счёт по своим событиям (сообщения/реакции/сайты/дни). Одномерно.
- ⬜ Многомерная (вклад/конструктивность/экспертиза/управление), per-community.
- ⬜ Сигналы: лайки/награды/флаги; сибил-защита (порог, охлаждение).
- ⬜ Профиль чужого участника (просмотр репутации, как «Люди» в расширении).

## 8. Управление (Governance) — §06, §03.1.7
- ⬜ Предложения (proposal-noet) и обсуждение.
- ⬜ Голосование (vote-noet), вес = функция репутации; подсчёт на клиенте.
- ⬜ Модерация (скрыть/удалить/бан) подписанными действиями по правилам.
- ⬜ Разделение властей, арбитраж, исполнение решений.

## 9. Реалтайм (Realtime) — §04.7
- 🟡 Звонок host-star + демонстрация экрана сделан в расширении; **порт в зону** нужен.
- ⬜ Группа >5 через форвард-узел из участников; запись (IPFS); стрим (WebTorrent §08.5).

## 10. Поиск и навигация
- ✅ Поиск-главная (имена/профили/страницы), переход на сайт.
- ⬜ Поиск сообществ и контента; теги/категории; «Люди» (каталог участников).

## 11. Узел (Node) — бэкбон
- ✅ Локальный шлюз (зона + API подписи + прокси картинок), кроссплатформа Tauri.
- ✅ Самообновление (tauri-updater; для релизов включить артефакты + секреты подписи).
- ⬜ Целевое: libp2p (DHT/GossipSub) вместо реле, NAT-traversal, фоновый P2P (§04.2-4).

## Очередь (что делаю дальше, по ценности)
1. **Фикс багов** (аватар-прокси, тег) — закрыть текущее.
2. **Порт из расширения в зону:** вики, «Люди» (каталог + чужой профиль/репутация), звонок.
3. **Личные сообщения (DM, E2EE)** и нормальные каналы/группы как сущности.
4. **Сообщества** как сущность (вступление, роли, конституция) + форк.
5. **Управление:** предложения + голосование с репутацией.
6. **Хранилище:** загрузка медиа в IPFS из узла.
7. **Realtime в зоне:** звонок/демонстрация/группа.
8. **Цель §04:** миграция сети на libp2p, логов на Hypercore, совместного на Yjs.
