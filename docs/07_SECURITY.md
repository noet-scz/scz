# noet Security: Модель угроз и защиты

> Этот документ описывает полную модель угроз для noet: от сетевых атак до государственных, от Sybil до атак на репутацию. Каждая угроза оценена по влиянию и вероятности, описаны митигации.

---

## Методология оценки

Каждая угроза оценивается по двум осям:

| Оценка | Влияние (Impact) | Вероятность (Likelihood) |
|---|---|---|
| **Критическое** | Паралич всей сети или массовая потеря данных | >50% в течение года |
| **Высокое** | Поражение отдельных сообществ или узлов | 10-50% в течение года |
| **Среднее** | Деградация сервиса, локальные проблемы | 1-10% в течение года |
| **Низкое** | Незначительные нарушения, теоретические | <1% в течение года |

## 1. Sybil-атаки

### 1.1. Описание

Создание множества фейковых идентичностей для манипуляции голосованиями, накрутки репутации, спама.

### 1.2. Векторы атаки

**Массовая регистрация:**
- Ботнет создаёт тысячи ключей (ключ = идентичность в noet)
- Голосует за/против предложений
- Накручивает репутацию "своим"

**Кольцевая накрутка:**
- N аккаунтов создают контент
- N+1..N+N аккаунтов лайкают его
- Репутация искусственно завышается

### 1.3. Оценка

| Параметр | Значение |
|---|---|
| Влияние | Высокое (искажение управления) |
| Вероятность | Высокая (создание ключей бесплатно) |
| Сложность атаки | Низкая |

### 1.4. Митигации

**Репутационный порог (Reputation Gating):**
- Новые аккаунты не имеют права голоса
- Минимальная репутация для участия в governance
- Период "испытательного срока" (например, 30 дней активности)

**Proof of Work для новых аккаунтов:**
- Создание аккаунта требует вычислений (NIP-13 в Nostr)
- Дорого для массового создания, дёшево для одного пользователя

**Кольцевой анализ:**
- Статистический анализ графа взаимодействий
- Кольцевая накрутка выявляется паттернами
- Автоматический downvote подозрительных связей

**Капча и rate limiting:**
- Для релей/узлов: ограничение регистраций с одного IP
- Proof-of-humanity (опционально, через сторонние сервисы)

### 1.5. Остаточный риск

При определённых затратах Sybil-атака возможна. Но стоимость атаки на одно сообщество не масштабируется на все сообщества. Защита на уровне сообщества, не протокола.

## 2. Спам

### 2.1. Описание

Массовая рассылка нежелательного контента: рекламы, фишинга, мемов, провокаций.

### 2.2. Векторы атаки

- Массовая публикация noet через открытые узлы
- Заполнение ленты мусором
- Флуд комментариями

### 2.3. Оценка

| Параметр | Значение |
|---|---|
| Влияние | Среднее (деградация UX) |
| Вероятность | Высокая |
| Сложность атаки | Очень низкая |

### 2.4. Митигации

**Экономическая:**
- Proof-of-work для публикации (опционально, configurable per community)
- Staking: публикация требует заморозки средств (slashing за спам)

**Репутационная:**
- Спам от низко-репутационных аккаунтов скрывается по умолчанию
- "Показывать только от аккаунтов с репутацией > X"
- Пользовательский фильтр

**Социальная:**
- Флаги от участников → модераторы review
- Reputation staking: "я ставлю свою репутацию, что это спам"

**Техническая:**
- Rate limiting на уровне узла
- Content-based фильтры (ML-based, локально)

## 3. DDoS / Flood-атаки

### 3.1. Описание

Перегрузка узлов сети фальшивым трафиком.

### 3.2. Векторы атаки

- UDP flood на DHT-узлы
- SYN flood на TCP-соединения
- GossipSub flood (массовая публикация в топики)
- Resource exhaustion (массовые запросы на хранение)

### 3.3. Оценка

| Параметр | Значение |
|---|---|
| Влияние | Среднее-Высокое (узлы выходят из строя) |
| Вероятность | Средняя |
| Сложность атаки | Средняя |

### 3.4. Митигации

**libp2p resource manager:**
- Лимиты на количество соединений, потоков, память
- Автоматическое отключение при превышении

**Connection scoring:**
- GossipSub score для пиров: полезные пиры повышаются, flood-еры понижаются
- Prune пиров с низким score

**DHT hardening:**
- Rate limiting на RPC-вызовы
- Проверка корректности запросов
- Routing table protection (Sybil-resistant DHT)

**Географическое распределение:**
- Узлы распределены по миру
- Атака на один регион не обрушивает сеть

**Proof-of-work для дорогих операций:**
- Публикация в DHT требует PoW
- Отклик на запрос требует PoW

## 4. Государственные атаки (State-level)

### 4.1. Описание

Атаки от государственных структур: цензура, блокировки, юридическое давление, субверсия.

### 4.2. Векторы атаки

**Блокировка инфраструктуры:**
- Блокировка IP-адресов узлов на уровне провайдеров
- Запрет на использование протокола
- Блокировка bootstrap-списков

**Юридическое давление:**
- Иски против разработчиков
- Требование удалить контент
- Обвинения в пособничестве

**Субверсия:**
- Внедрение агентов в сообщества
- Создание провокационного контента для дискредитации
- Финансирование раскола

**Массовый анализ:**
- Пассивный мониторинг трафика
- Корреляция публичных ключей с реальными личностями
- Анализ метаданных

### 4.3. Оценка

| Параметр | Значение |
|---|---|
| Вияние | Критическое (если успешно) |
| Вероятность | Средняя (зависит от юрисдикции) |
| Сложность атаки | Высокая (для полной остановки) |

### 4.4. Митигации

**Архитектурные:**

- **Нет центральных серверов** — блокировать нечего
- **DHT-based discovery** — узлы находят друг друга без центрального реестра
- **P2P-only режим** — сообщество может работать в полном P2P, без публичных узлов
- **Bootstrap через множество каналов** — DNS, IRC, другие сети, dead drops

**Криптографические:**

- **Весь трафик зашифрован** — невозможно отличить noet-трафик от другого
- **Padding** — фиксированный размер пакетов, не выдаёт тип контента
- **Plausible deniability** — любой узел может сказать "я только ретранслирую"

**Социальные:**

- **Распределённая разработка** — разработчики в разных юрисдикциях
- **Анонимность ключевых участников** — не обязательна, но возможна
- **Международное сообщество** — атака на одну юрисдикцию не останавливает всё

**Протокольные:**

- **Domain fronting** — маскировка под популярные сервисы
- **Steganography** — встраивание в другие протоколы
- **Bridge nodes** — специальные узлы для обхода блокировок

### 4.5. Сценарий: Полная блокировка страной X

```
1. Страна X блокирует известные IP узлов
   → Узлы меняют IP (динамические, VPN, proxy)
   
2. Страна X блокирует DHT-трафик по сигнатуре
   → noet переключается на маскировку (TLS, domain fronting)
   
3. Страна X запрещает noet законом
   → Пользователи используют bridge nodes
   → Или переходят в полный P2P (только direct connections)
   → Или используют steganography (скрытый в других протоколах)
   
4. Страна X преследует пользователей
   → noet не требует телефона/email (анонимность)
   → Трафик неотличим от другого зашифрованного
   → Plausible deniability для relay-узлов
```

**Вывод:** Полная блокировка noet в стране требует полного запрета на шифрованный P2P-трафик. Это возможно теоретически, но на практике означает блокировку всего интернета.

## 5. Враждебные форки (Hostile Forks)

### 5.1. Описание

Форк сообщества с целью захвата бренда, участников, дискредитации оригинала.

### 5.2. Векторы атаки

- **Fork + rebrand** — форк представляет себя как "истинный" оригинал
- **Fork + poison** — форк публикует вредоносный код под именем оригинала
- **Множественные форки** — создание десятков форков для путаницы

### 5.3. Митигации

- **Криптографическая идентичность** — оригинал подписан ключом основателя
- **Первооткрывательство** — timestamp в DHT или блокчейне доказывает, кто первый
- **Репутационное наследие** — участники знают историю, доверяют оригиналу
- **Невозможность отмены форка** — форк всегда возможен, но легитимность определяется сообществом

## 6. Захват инфраструктуры

### 6.1. Описание

Контроль над критической инфраструктурой: bootstrap-узлами, popular relays, domain names.

### 6.2. Векторы

- Контроль над популярными узлами (большинство трафика через них)
- Захват доменов (если используются)
- Компрометация кода (вредоносный PR)

### 6.3. Митигации

- **Bootstrap диверсификация** — списки bootstrap распространяются множеством каналов
- **No single popular node** — DHT не имеет "популярных" узлов, все равноправны
- **Code signing + reproducible builds** — любой может проверить, что код не модифицирован
- **Множественные клиенты** — атака на один клиент не атакует всех
- **Domain-free** — noet не зависит от доменов (DHT-based discovery)

## 7. Атаки на репутацию

### 7.1. Описание

Манипуляция системой репутации для дискредитации участников или завышения своей репутации.

### 7.2. Векторы

**Reputation bombing:**
- Координированная массовая оценка "дизлайк" целевому участнику
- Цель: исключение из governance, потеря веса голоса

**Reputation farming:**
- Создание контента, оптимизированного под метрики репутации
- Low-quality, но high-engagement контент

**Reputation selling:**
- Продажа аккаунтов с высокой репутацией
- Передача ключей = передача репутации (где это технически возможно)

### 7.3. Митигации

**Многомерность:**
- Репутация — не одно число, а вектор
- Атака на одну метрику не уничтожает всю репутацию

**Ограничение скорости:**
- Оценки от одного участника имеют diminishing returns
- Нельзя за день изменить репутацию на порядок

**Социальная проверка:**
- Оценки от "близких" в графе доверия имеют больший вес
- Оценки от "далёких" — меньший

**Непередаваемость:**
- Продажа аккаунта = продажа приватного ключа
- Это возможно технически, но легко обнаруживается (резкая смена поведения)
- Репутация привязана к паттернам поведения, не только к ключу

## 8. Атаки на шифрование

### 8.1. Квантовые компьютеры

**Угроза:** Квантовый компьютер способен взломать secp256k1, Ed25519, RSA.

**Временной горизонт:** 10-30 лет.

**Митигация:**
- Roadmap на пост-квантовые подписи (SPHINCS+, Falcon)
- Гибридные схемы (PQ + classical) на переходный период
- Агрессивное тестирование PQ-криптографии

### 8.2. Реализационные баги

**Угроза:** Уязвимость в криптографической библиотеке.

**Митигация:**
- Использование проверенных библиотек (libsecp256k1, ring, libsodium)
- Аудиты безопасности
- Reproducible builds
- Bug bounty программа

## 9. Атаки на доступность

### 9.1. Eclipse attack (на DHT)

**Описание:** Атакующий окружает целевой узел в DHT своими зловредными узлами, контролируя всю маршрутизацию.

**Митигации:**
- S/Kademlia: PoW для node ID
- Multi-path lookups
- Bootstrap через доверенные каналы
- Периодический audit routing table

### 9.2. Churn attack

**Описание:** Массовый вход и выход узлов для дестабилизации DHT.

**Митигации:**
- Stabilization period: новый узел не участвует в routing N минут
- Reputation for peers: стабильные узлы предпочитаются
- Aggressive caching

## 11. Security Roadmap

### Фаза 1: Базовая безопасность (запуск)

- [ ] Аудит криптографических примитивов
- [ ] Resource manager на всех узлах
- [ ] GossipSub scoring enabled
- [ ] Bootstrap diversification
- [ ] Rate limiting

### Фаза 2: Укрепление (6-12 месяцев)

- [ ] Sybil-resistant DHT (PoW for node IDs)
- [ ] Reputation system v1
- [ ] Automated abuse detection
- [ ] Bug bounty программа
- [ ] Security advisory process

### Фаза 3: Зрелость (1-2 года)

- [ ] Formal verification критических компонентов
- [ ] PQ-cryptography research
- [ ] Penetration testing
- [ ] Decoy nodes (honeypots) для обнаружения атак

### Фаза 4: Долгосрочная устойчивость (3-5 лет)

- [ ] Пост-квантовая миграция (при необходимости)
- [ ] Диверсификация криптографии
- [ ] Red teaming (постоянное тестирование)

## 12. Заключение

Безопасность noet — это не состояние, а процесс. Угрозы эволюционируют, и защита должна эволюционировать вместе с ними.

Ключевые принципы:
1. **Defense in depth** — многослойная защита
2. **Fail safe** — при отказе система безопасна
3. **Minimize trust** — не доверяй, проверяй
4. **Assume breach** — готовься к компрометации отдельных компонентов
5. **Open design** — безопасность не в секретности, а в криптографии
