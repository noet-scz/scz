# 02 — Примитивы

Всё в noet сводится к пяти примитивам. Новых сущностей в ядре нет — есть только эти пять
и их композиции. Если что-то можно выразить композицией — отдельный механизм не создаём.

Берём готовое (концепция §4): где есть подходящий NIP — используем его, свой kind минтим
только когда ничего не подходит.

---

## 1. Identity — криптографическая личность

**Что это:** пара ключей Nostr. npub это и есть человек/агент. Ни почты, ни аккаунта на
чужой машине.

**Несёт:** ключи, профиль, делегирование прав, поручительства, доверие, связи, репутацию.

**В событиях:**
| Смысл | kind | Источник |
|---|---|---|
| Профиль (имя, аватар, о себе) | `0` | NIP-01 |
| Контакты / подписки | `3` | NIP-02 |
| Подпись действия (вход) | `22242` | NIP-42 |
| Поручительство / отзыв | `8001` / `8002` | noet (gov.mjs) |
| Доступ / отзыв доступа | `8010` / `8011` | noet |
| Запрос инвайта (одноразовый) | `8020` | noet |

Репутация это **детерминированная функция** над публичными событиями (поручительства +
стаж + живые артефакты), а не хранимое число. Любой клиент считает её одинаково. Гейтит
ёмкость, не власть (концепция §5).

**Ключи прячем за UX:** ключ живёт в расширении (window.nostr / NIP-07), подпись локальна.
Восстановление: бэкап-файл сейчас; паскеи и социальное восстановление — на роадмапе.

---

## 2. Space — универсальное пространство

**Что это:** контейнер с владельцем и политикой. Одна сущность покрывает сообщество,
канал, форум, проект, группу, организацию, каталог, вики, магазин, личную страницу.

**Тип это не код ядра, а поле.** `kind:31000` (адресуемое, NIP-33), `d`-тег = id
пространства, content = `{ type, title, about }`, где `type ∈ {page, feed, forum, board,
wiki, catalog, market, …}` — **строка, которую читает шаблон снаружи, а не ядро.**

```json
{ "kind": 31000, "tags": [["d","alice-blog"], ["name","Блог Алисы"]],
  "content": "{\"type\":\"feed\",\"title\":\"Блог Алисы\",\"policy\":\"alice-blog\"}" }
```

Владелец = pubkey автора события. Имя (`alice.me`) указывает на координату этого Space
(см. реестр в [04-architecture.md](04-architecture.md)). Личная страница `.me` — частный
случай Space с `type:"page"`.

Базируемся на NIP-72 (moderated communities, kind 34550) как на родственнике, но обобщаем
поле `type`, чтобы один примитив покрывал все структуры.

---

## 3. Artifact — универсальный объект данных

**Что это:** единица содержимого. Сообщение, файл, статья, картинка, видео, документ,
задача, событие, голосование — всё это Artifact с разным `type`/kind.

**Правило размера:**
- **мелкое/социальное** (пост, коммент, реакция) → событие Nostr целиком;
- **крупное/бинарное** (файл, картинка, собранный сайт) → в IPFS, в событии лежит CID.

| Смысл | kind | Источник |
|---|---|---|
| Короткий пост | `1` | NIP-01 |
| Лонгрид (markdown) | `30023` | NIP-23 |
| Файл (метаданные + CID) | `1063` | NIP-94 |
| Страница/сайт (изменяемая, → IPFS CID) | `31002` адрес. | noet |
| Произвольный артефакт приложения | `30078` (param) | NIP-78 |

Страница `.me` сегодня = запись в реестре `name → CID`. Цель: страница это адресуемый
Artifact автора (`kind:31002`, `d`=имя, content содержит CID), а реестр лишь сопоставляет
человекочитаемое имя этой координате. Тогда страница переносима и не привязана к серверу.

## 4. Relation — связи между сущностями

**Что это:** ребро графа. Ссылка, ответ, цитата, подписка, принадлежность, доверие,
упоминание, зависимость. **Связи — это теги, не отдельные сущности.** «Лайк», «репост»,
«закладка», «подписка» в ядре не существуют как фичи; это Relation с разным типом.

| Связь | Как | Источник |
|---|---|---|
| Ответ / ветка | `e`-теги (root/reply) | NIP-10 |
| Упоминание | `p`-тег | NIP-01 |
| Принадлежит Space | `a`-тег (координата 31000) | NIP-33 |
| Тема / тег | `t`-тег | NIP-12 |
| Реакция (лайк) | `kind:7` | NIP-25 |
| Цитата / репост | `q`-тег / `kind:6` | NIP-18 |
| Доверие | `8001` (Identity) | noet |
| Произвольная (цитирование, зависимость) | `rel`-тег `["rel","cites",<id>]` | noet |

Форум, лента, граф знаний, дерево задач — это разные **прочтения одного графа связей**,
а не разные движки.

---

## 5. Policy — правила взаимодействия

**Что это:** машиночитаемые правила Space или всего экземпляра. Кто пишет, кто модерирует,
какие лимиты, условия хранения, как меняется само правило.

**`kind:31001`** (адресуемое), `d` = scope (id пространства или `*` для экземпляра),
content = правила:

```json
{ "kind": 31001, "tags": [["d","alice-forum"]],
  "content": "{\"post\":{\"minRep\":3},\"mods\":[\"npub…\"],\"moderation\":\"post-approval\"}" }
```

Наши делегации/поручительства (gov.mjs) — это Policy на уровне экземпляра. Обобщение:
**та же машинерия доступов/репутации работает на уровне Space** через `a`-тег. Владелец
Space выдаёт модерацию внутри своего Space ровно теми же событиями `8010`, scope =
`space:<id>`. Один механизм — и для экземпляра, и для любого пространства внутри.

Модерация по NIP-72: модератор Space публикует событие-одобрение/удаление; клиенты чтят
решения модераторов своего Space. Удаление — NIP-09 (`kind:5`).

## North Star: всё собирается из пяти

| Продукт | Space.type | Artifact | Relation | Policy |
|---|---|---|---|---|
| **Блог** | feed | посты (30023) | ответы (e), темы (t) | только автор пишет |
| **Форум** | forum | темы и посты (1) | ветки (e), принадлежность (a) | minRep на постинг, модераторы |
| **Чат** | feed | сообщения (1, ephemeral) | упоминания (p) | участники Space |
| **Доска/канбан** | board | задачи (30078) | зависимости (rel), колонки (t) | члены проекта правят |
| **Вики** | wiki | статьи (31002→CID) | ссылки (rel "links"), версии | редакторы с доступом |
| **Маркетплейс** | market | лоты (30078 + CID фото) | категории (t), отзывы (e) | продавец с репутацией N |
| **Каталог/медиатека** | catalog | файлы (1063→CID) | коллекции (a), теги (t) | владелец коллекции |
| **Соцсеть** | feed | посты + профили (0) | подписки (3), лайки (7) | открытая или по инвайту |

**Ни одна строка не требует нового механизма ядра.** Меняется только шаблон-рендерер
снаружи (в IPFS, открывается расширением). Это и есть конструктор: ядро из пяти примитивов,
бесконечность продуктов поверх.

## SDK как поверхность конструктора

`window.noet` (инжектится расширением на каждую страницу) даёт примитивы напрямую:

```js
noet.identity.me()                     // кто я
noet.space.create({ type, title })     // создать пространство
noet.space.get(coord)                  // прочитать
noet.artifact.publish({ space, … })    // добавить артефакт (мелкое→событие, крупное→IPFS)
noet.artifact.query({ space, filter }) // прочитать ленту/граф
noet.relation.add(from, kind, to)      // связь (ответ, лайк, принадлежность…)
noet.policy.set(scope, rules)          // правила пространства
noet.policy.can(pubkey, action, scope) // проверка права (клиентская)
```

Разработчик собирает форум/магазин/вики этими вызовами, **не трогая ядро и не поднимая
сервер**. Проверка для любой новой возможности SDK: использует существующий примитив? да →
ок; нет → это скрытый монолит, пересмотреть.
